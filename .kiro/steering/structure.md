# Project Structure

## Layout

```
.
├── .amazonq/rules/         # Amazon Q coding rules (common, GitHub Actions, Terraform/AWS)
├── .kiro/steering/         # Kiro steering documents
├── .pre-commit-config.yaml # Pre-commit hook configuration
└── README.md               # Project documentation with live coding prompts
```

Terraform infrastructure is organized into root modules per concern (e.g., `github-oidc/`, `ecs-nginx/`), each with its own state.

## Terraform File Conventions

Each root module follows this file layout:

| File | Purpose |
|------|---------|
| `providers.tf` | AWS provider configuration |
| `versions.tf` | Terraform version and provider version constraints |
| `backend.tf` | S3 remote state backend configuration |
| `main.tf` | Core resources (split by service when possible) |
| `variables.tf` | Input variables with descriptions and validations |
| `outputs.tf` | Output values (no sensitive data) |
| `README.md` | Auto-generated by terraform-docs |
| `.terraform.lock.hcl` | Provider lock file (committed to git) |

## Naming and Tagging

- Resource names: no type suffix/prefix (use `my-app`, not `my-app-vpc`)
- Required tags on all resources: `Environment`, `Owner`, `Project`, `Cost Center`, `Managed By: terraform`, `Root Module URL`

## Networking Pattern

- Private subnets: ECS tasks, databases, internal services
- Public subnets: ALB, NAT Gateways
- No direct internet access for compute resources
